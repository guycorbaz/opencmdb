# Story 4.1: Freeze the JSONL observation stream

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the engine's test infrastructure,
I want the on-disk fixture format frozen as a committed JSONL stream of `Observation`s, resolved through a single path constant and locked by a sha256 manifest,
so that every trap that follows is written against a format that cannot drift, cannot be regenerated by the build of the code under test, and cannot leak a real network.

## Context

Epic 4 builds the substrate the identity engine (Epic 5) is driven against: ~50 adversarial traps, a `FixtureConnector` that replays them, and a metrics harness. **This story freezes only the observation stream** — the bottom layer everything else stands on. It is deliberately small.

D19 decides the schema for us and the decision is not reopenable: *"the fixture is a serialised stream of Observations + a truth labelling. The `Connector`'s job is to emit Observations. **Therefore the fixture schema IS the Observation schema.**"* The types already exist and already round-trip through serde. **This story writes a file, a path constant, a reader and a hash — not a schema.**

**Explicitly OUT of scope** (do not start these):
- The **truth labelling / expectation format** (`expect_rule`, the mandatory one-sentence `reason`, `MustMerge`/`MustNotMerge`/`MustAbstain`) → next story.
- **`FixtureConnector`** itself → a later story; it must pass `run_connector_contract`.
- The **seeded generator**, the `recapture` tool, the metrics harness, any actual trap.
- Migrating `fixtures/MANIFEST` to the D56 `MANIFEST.toml` (sha256 + seed + generator version). The provisional line format stays for now; the migration is its own story once a generator exists to have a version.
- **Orphan-fixture detection** — a file present in `fixtures/` but absent from the MANIFEST is still not caught (deferred from the story-1.2 review). It becomes meaningful for the first time once this story puts real files on disk, and it belongs with the MANIFEST.toml migration that will walk the tree anyway. Left open deliberately, not forgotten.

## Acceptance Criteria

1. **`fixtures/` exists at the workspace root** with two subdirectories, `scenario/` and `capture/`, each carrying a short `README.md` stating what belongs there and — for `capture/` — that the future re-capture job must never be able to see `scenario/` (D56: a re-capture that could reach a synthetic trap would rewrite truth to make the gate pass).

2. **One committed scenario fixture** at `fixtures/scenario/replay/<name>.jsonl` holds **one serialized `Observation` per line**, in the types' existing serde representation. No new schema, no DTO, no hand-rolled format, no wrapper object.

3. **Every value in the fixture is synthetic and stable.** IPv4 addresses from RFC 5737 (`192.0.2.0/24`), locally-administered MACs, invented hostnames. `obs_id`, `connector_id`, the `scope` ids and `observed_at` are **fixed literals in the file** — never generated at read time. A `Uuid::now_v7()` or a clock read anywhere near this fixture is a defect: D19 requires `obs_id` stable so truth can point at it, and `observed_at` to come from the fixture so the engine never touches the clock.

4. **The fixtures root is resolved from exactly one constant in the whole tree**, per D56: `concat!(env!("CARGO_MANIFEST_DIR"), "/../../fixtures/")`. A second occurrence anywhere means it is already broken.

5. **A reader parses a `.jsonl` into `Vec<Observation>`** and fails with the **1-indexed line number** on a malformed line — never silently skipping one. It lives in `opencmdb-bin` (D47: the domain crate cannot read files).

6. **A test reads the committed fixture and asserts the exact expected `Vec<Observation>`**, constructed literally in the test. Re-serializing that vector reproduces the committed file **byte for byte** (trailing newline included) — so a future serde rename cannot silently change what the corpus means.

7. **The gate stops being vacuous.** `fixtures/MANIFEST` lists the fixture with its real sha256, and `cargo xtask ci` reports `1 fixture(s) match their recorded sha256` instead of `no fixtures — skipped`.

8. **All gates green:** `cargo fmt --all`, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo test --workspace`, `cargo xtask ci`.

## Tasks / Subtasks

- [ ] **Task 1 — the fixtures tree** (AC: 1)
  - [ ] Create `fixtures/scenario/replay/` and `fixtures/capture/`.
  - [ ] `fixtures/README.md`: what this directory is (a spec that precedes the code, not test data — D56/D45), and that editing a file here is a commit that says *"I am changing the spec"*.
  - [ ] `fixtures/capture/README.md`: real, version-tagged, dated captures prove the **parser**; they rot and get re-captured. `fixtures/scenario/README.md`: synthetic, written to trap X, prove the **engine**; they do **not** rot — they are right or wrong, and the re-capture job must never reach them.

- [ ] **Task 2 — write the fixture** (AC: 2, 3)
  - [ ] Choose a small, boring content: 2–3 observations, at least one carrying more than one `Fact`, at least one `Mac` and one `IpV4` so the format is exercised beyond a single variant.
  - [ ] Serialize by hand or by a throwaway test print — but **commit the file**; nothing generates it at build or test time (D19: the artefact is versioned; a generator bug must never rewrite truth silently).
  - [ ] Verify every address is RFC 5737 and every MAC is locally-administered. **No real network data, ever** — D19 calls real captures in a public repo *"a privacy liability … disqualifying. Not debatable."*

- [ ] **Task 3 — path constant + reader** (AC: 4, 5)
  - [ ] New module `crates/opencmdb-bin/src/fixtures.rs`: the single `FIXTURES_DIR` constant and `read_jsonl(path) -> Result<Vec<Observation>, …>`.
  - [ ] Error carries the 1-indexed line number and the underlying serde message.
  - [ ] Register the module in `main.rs` (alphabetical: after `dburl`, before `metrics`).

- [ ] **Task 4 — tests** (AC: 6)
  - [ ] Read the committed fixture; assert equality against a literally-constructed `Vec<Observation>`.
  - [ ] Re-serialize and assert the bytes equal the committed file exactly.
  - [ ] A malformed line yields an error naming the right line number.

- [ ] **Task 5 — lock it** (AC: 7, 8)
  - [ ] `sha256sum` the fixture from inside `fixtures/` so the path is relative, and append the line to `fixtures/MANIFEST` (format: `<sha256>␠␠<path>`, exactly what `sha256sum` prints).
  - [ ] Run `cargo xtask ci` and confirm the fixtures line now reports a real count.
  - [ ] Run the full gate set.

## Dev Notes

### The schema already exists — do not invent one

`Observation`, `Fact`, `Scope`, `Capabilities`, `HostnameSource`, `FactKind` and `MacAddr` all already derive `Serialize`/`Deserialize` [Source: crates/opencmdb-core/src/observation/mod.rs:71,128,140,173,205,215,232]. A serde round-trip is already tested there (`observation_serde_roundtrips`, line 362). **The JSONL line is `serde_json::to_string(&observation)`. There is nothing to design.** Writing a parallel DTO, a custom `Serialize`, or a "fixture format" struct would be the exact wheel-reinvention this story exists to prevent.

`serde_json` is already a dependency of `opencmdb-bin` — no new crate is needed.

### The architecture document is OLDER than the code — trust the code

Two traps for anyone reading `architecture.md` literally:

- **The `Connector` trait shown in D19 is stale.** It reads `fn poll(&mut self, now) -> Result<Vec<Observation>, ConnectorError>` plus `fn capabilities(&self)`. The real trait is `async fn poll(&mut self, now, sink: &mut dyn ObservationSink, cancel: CancellationToken) -> Result<PollSummary, ConnectorError>`, with capabilities reported inside `PollSummary` [Source: crates/opencmdb-core/src/connector/mod.rs:128-140]. This story does not implement the trait, but do not "correct" anything toward the old signature.
- **Every mention of SQLite is dead** (D64: MariaDB is the only engine). `architecture.md` still names `SqliteRepository` in places; ignore it.

### The path constant is a D56 non-negotiable

> *"There is no `CARGO_WORKSPACE_DIR`. The path is `concat!(env!("CARGO_MANIFEST_DIR"), "/../../fixtures/")` — a single constant, in one module, never copied. If it appears more than once in the tree, it is already broken."* [Source: _bmad-output/planning-artifacts/architecture.md#D56]

From `crates/opencmdb-bin`, `../../fixtures/` resolves to the workspace root. Do not use `std::env::current_dir()` (differs between `cargo test` and a running binary) and do not add a `build.rs` — D56 notes a `build.rs` here is one step from letting the build regenerate the oracle.

### Why `fixtures/` sits at the root, and why that matters to how you write it

D56, quoting D45: a file under `tests/` is read as the property of the test, and *"the first reflex of someone refactoring the engine is to adjust them until the red goes away. A red repairable by editing the spec is not a gate — it is a negotiation."* The READMEs in Task 1 are not decoration; they are what makes the next person hesitate.

### The gate already exists, and its prove-to-red tests already exist

`gate_fixture_manifest` [Source: xtask/src/main.rs:397-433] reads `fixtures/MANIFEST`, recomputes each sha256 and fails closed. Its unit tests already cover sha mismatch, byte match, case-insensitive comparison, a missing file, malformed lines and the no-fixtures case [Source: xtask/src/main.rs:760-845]. **Do not write new gate tests for those paths.** The only change this story makes to the gate is that it stops being vacuous, because a real fixture is finally listed.

The provisional MANIFEST line format is `<sha256-hex>  <path>`, path relative to `fixtures/`, blank lines and `#` comments skipped, anything else `Malformed` [Source: xtask/src/main.rs:445-470]. `sha256sum` output matches this exactly — run it with `fixtures/` as the working directory so the recorded path is relative.

### Privacy is a hard constraint here, not a preference

D19: real captures are *"a privacy liability in a public repo — MACs, hostnames (`guy-macbook`), the topology of Guy's home. For an open-source project this is disqualifying. Not debatable."* This repository is public. Use RFC 5737 (`192.0.2.0/24`) addresses and locally-administered MACs (the `locally_administered` flag on `Fact::Mac` exists for exactly this).

### Determinism is the whole point

`obs_id` stable so truth can point at it; `observed_at` from the fixture so the engine never touches the clock; *"DHCP churn is tested by replaying timestamps"* [Source: architecture.md#D19]. `FixedClock` already exists for the Rust-side equivalent [Source: crates/opencmdb-core/src/testing/mod.rs:16-23]. Nothing in the read path may consult the wall clock or mint an id.

### Project Structure Notes

- **`FixtureConnector` will live in `crates/opencmdb-bin/src/connectors/` per D56** — but the crate is currently flat (`arp_ping.rs`, `auth.rs`, `dburl.rs`, `metrics.rs`, `page.rs`, `repo.rs`). **Do not create `connectors/` in this story**; introducing it while moving `arp_ping.rs` is churn that belongs with the story that actually adds the second connector. This story adds a flat `fixtures.rs`, consistent with what is there.
- `#![allow(dead_code)]` is the established pattern for a module wired in ahead of its consumer — see the head of `arp_ping.rs`. Use it rather than inventing a `#[cfg(test)]`-only module: the reader is production code that the `FixtureConnector` will call.
- Module declarations in `main.rs` are alphabetical [Source: crates/opencmdb-bin/src/main.rs:9-14].

### Testing standards

- Unit tests live in a `#[cfg(test)] mod tests` at the foot of the module, the convention throughout this crate.
- Tests that read the fixture are ordinary tests — no feature gate, no env gate. The file is committed, so they run everywhere including CI.
- Run the full local gate before declaring done: `cargo fmt --all && cargo clippy --workspace --all-targets -- -D warnings && cargo test --workspace && cargo xtask ci`. Epic 3's retrospective recorded four CI-only failures caused by skipping `--all-targets` or `xtask ci` locally.

### Previous story intelligence (3.10 and the 0.1.1 deployment work)

- The `views-hash` gate reports `ℹ STALE` and that is expected — `architecture-views.md` is regenerated at milestones, not per-change. It does not fail the build. Do not "fix" it.
- Recent work added `futures-util` and `percent-encoding` to `opencmdb-bin`. **No new dependency is needed for this story**; if you reach for one, re-read the first Dev Note.
- The dependency-frontier gate is a reflex gate with a known blind spot for optional/feature-gated edges (issue #2). Adding anything to `opencmdb-core` here would be out of scope anyway — the reader belongs in `bin`.
